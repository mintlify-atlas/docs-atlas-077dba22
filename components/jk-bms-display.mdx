---
title: "JK-BMS Display Port Monitor for ESPHome (RS485)"
description: "Passively monitor a JK-BMS over the RS485 display port with ESPHome. Read-only access to cell voltages, pack state, temperatures, and protection flags."
---

The `jk_bms_display` component **passively monitors** a JK-BMS by tapping into the data stream on the **RS485 display port** — the same connector used by the BMS's own LCD panel. It is entirely read-only: no commands are ever sent to the BMS, making it a safe monitoring option that does not interfere with BMS operation.

Because data is broadcast by the BMS periodically to drive the display, you only need to listen on the RS485 bus. An RS485-to-TTL adapter converts the differential signal to the UART levels accepted by the ESP32.

<Note>
  Requires ESPHome **2024.12.0** or newer. The component auto-loads `binary_sensor` and `sensor`. `MULTI_CONF = True` supports multiple BMS units on separate UART ports.
</Note>

## Supported Hardware

This component is compatible with JK-BMS models that have a dedicated display connector, including:

- **JK-BD6A2xS10P**
- **JK-BxA24SxxP**

Other models with a 6-pin display port using RS-485 should also work.

## Wiring

The display port provides a differential RS-485 signal. An RS-485-to-TTL module bridges the signal to the ESP32 UART RX pin. The port runs at **2400 baud**. No TX connection is needed — this is receive-only.

```
JK-BMS display port (6-pin)          RS-485-to-TTL module     ESP32
─────────────────────────────────────────────────────────────────────
Pin A  ──────────────────────────── A+                GND ─── GND
Pin B  ──────────────────────────── B-                RXD ─── RX pin
Pin VCC ─────────────────────────── VCC               VCC ─── 5V
Pin GND ─────────────────────────── GND                        
```

<Warning>
  Do **not** connect TXD from the RS-485 adapter to any ESP32 pin. This is a passive monitor — any accidental transmission could corrupt the BMS display data or cause unintended BMS behavior.
</Warning>

<Info>
  The K+ and K- pins on the display port are for the keypad input — they are not needed for this integration.
</Info>

## Configuration

<Steps>
  <Step title="Connect the RS-485-to-TTL adapter">
    Wire A+ and B- from the display port to the adapter, connect GND and 5V, and run the adapter's RXD to your chosen ESP32 GPIO.
  </Step>
  <Step title="Declare the UART bus">
    Configure the UART at 2400 baud with only an RX pin — no TX pin is required.
  </Step>
  <Step title="Declare jk_bms_display">
    Reference the UART bus by ID.
  </Step>
  <Step title="Add sensor and binary_sensor platforms">
    Each platform block references the `jk_bms_display` hub.
  </Step>
</Steps>

## Component Configuration

<ParamField path="uart_id" type="string" required>
  ID of the UART bus connected to the RS-485-to-TTL adapter. Must be configured at **2400 baud** with only an RX pin.
</ParamField>

<ParamField path="id" type="string">
  Optional ID used to reference this display monitor from sensor and binary_sensor platform blocks.
</ParamField>

<Note>
  This component is **read-only**. There are no switch, number, or text sensor platforms — all control must go through the `jk_bms` (UART/Modbus) or `jk_bms_ble` (Bluetooth) component instead.
</Note>

## Supported Platforms

### Sensor

Cell voltages `cell_voltage_1` through `cell_voltage_24` are available (V, 3 decimal places).

| Key | Unit | Notes |
|-----|------|-------|
| `total_voltage` | V | Total pack voltage |
| `current` | A | Pack current (positive = charge) |
| `power` | W | Total instantaneous power |
| `charging_power` | W | Power delivered during charging |
| `discharging_power` | W | Power consumed during discharging |
| `state_of_charge` | % | Battery state of charge |
| `min_cell_voltage` | V | Lowest cell voltage |
| `max_cell_voltage` | V | Highest cell voltage |
| `min_voltage_cell` | — | Index of lowest-voltage cell |
| `max_voltage_cell` | — | Index of highest-voltage cell |
| `delta_cell_voltage` | V | Voltage spread across cells |
| `average_cell_voltage` | V | Mean cell voltage |
| `mosfet_temperature` | °C | MOSFET/power tube temperature |
| `battery_temperature` | °C | Battery pack temperature |
| `cell_voltage_1` … `cell_voltage_24` | V | Individual cell voltages |

### Binary Sensor

All binary sensors carry the `diagnostic` entity category except `balancing_switch`, `charging`, and `discharging`.

| Key | Category | Notes |
|-----|----------|-------|
| `charging` | — | Charging MOSFET active |
| `discharging` | — | Discharging MOSFET active |
| `balancing_switch` | — | Balancer enabled |
| `system_warning` | diagnostic | General system warning flag |
| `cell_voltage_undervoltage_protection` | diagnostic | UVP triggered |
| `cell_voltage_overvoltage_protection` | diagnostic | OVP triggered |
| `overcurrent_protection` | diagnostic | OCP triggered |
| `mosfet_overtemperature_protection` | diagnostic | MOSFET OTP triggered |
| `battery_temperature_protection` | diagnostic | Battery OTP triggered |
| `short_circuit_protection` | diagnostic | SCP triggered |
| `coprocessor_communication_error` | diagnostic | Internal co-processor communication failure |
| `balancer_wire_resistance_too_high` | diagnostic | Balancer wire resistance fault |
| `cell_count_mismatch` | diagnostic | Detected cell count differs from configured |
