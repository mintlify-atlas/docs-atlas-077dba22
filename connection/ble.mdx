---
title: 'Monitor a JK-BMS wirelessly over Bluetooth LE (BLE)'
description: 'Set up wireless JK-BMS monitoring on ESP32 using jk_bms_ble. Covers protocol version selection, MAC address discovery, and OTA-safe BLE configuration.'
---

The JK-BMS advertises a BLE service over the standard `0xFFE0` / `0xFFE1` characteristic pair. The `jk_bms_ble` component connects to the BMS as a BLE client, subscribes to notifications, and sends periodic request frames to receive cell voltages, temperatures, protection thresholds, and writable settings — all without any physical serial wiring.

<Warning>
BLE is only supported on **ESP32** boards. The ESP8266 has no Bluetooth hardware and cannot use this connection method. Use the [UART-TTL connection](/connection/uart-ttl) if you own an ESP8266.
</Warning>

## Prerequisites

- An ESP32 board (any variant with BLE hardware)
- The MAC address of your JK-BMS (see [finding the MAC address](#finding-the-mac-address) below)
- The correct `protocol_version` for your hardware revision (see the table below)

## Choosing the right protocol version

The BMS firmware uses one of three frame formats depending on the hardware revision. Using the wrong version will result in garbled or absent sensor data.

| `protocol_version` | Hardware version | Example models |
|--------------------|-----------------|----------------|
| `JK04`             | ≤ 3.0           | JK-B2A16S hw 3.0, JK-B5A24S hw 8.x (old 4-byte protocol) |
| `JK02_24S`         | 6.0 – 10.x      | JK-B2A24S15P hw 8.x/10.x, JK-B2A24S20P hw 10.x |
| `JK02_32S`         | 11.x and newer  | JK-B2A8S20P hw 11.XW, JK-B2A20S20P hw 11.XW |

<Tip>
If you are unsure of your hardware version, check the sticker on the BMS or look up the version reported by the official Jikong Android app. When in doubt, start with `JK02_24S` (6.x–10.x) and switch to `JK02_32S` (11.x+) if cell resistance sensors or additional temperature sensors are missing from the decoded data.
</Tip>

## Finding the MAC address

The easiest way to find the BMS Bluetooth MAC address is with the `esp32_ble_tracker` component. Add the following minimal configuration, flash it, and watch the ESPHome logs for an advertisement from a device whose name starts with `JK`:

```yaml
esp32_ble_tracker:
  scan_parameters:
    active: false
```

The log will contain lines similar to:

```text
[esp32_ble_tracker] Found device C8:47:8C:E1:E2:AA RSSI=-63
[esp32_ble_tracker]   Name: 'JK-B2A8S20P'
```

Copy the MAC address shown and use it as `mac_address` in your configuration.

## Setup

<Steps>
  <Step title="Install the external component">
    ```yaml
    external_components:
      - source: github://syssi/esphome-jk-bms@main
        refresh: 0s
    ```
  </Step>

  <Step title="Select the protocol version and set the MAC address">
    Set the two substitution variables that drive the BLE configuration. Replace the MAC address with the one you found in the previous step.

    <Tabs>
      <Tab title="JK02_32S (hw 11.x+)">
        ```yaml
        substitutions:
          mac_address: C8:47:8C:E1:E2:AA   # replace with your BMS MAC
          protocol_version: JK02_32S
        ```
      </Tab>
      <Tab title="JK02_24S (hw 6.x–10.x)">
        ```yaml
        substitutions:
          mac_address: C8:47:8C:E1:E2:AA   # replace with your BMS MAC
          protocol_version: JK02_24S
        ```
      </Tab>
      <Tab title="JK04 (hw ≤ 3.0)">
        ```yaml
        substitutions:
          mac_address: C8:47:8C:E1:E2:AA   # replace with your BMS MAC
          protocol_version: JK04
        ```
      </Tab>
    </Tabs>
  </Step>

  <Step title="Configure the BLE stack and jk_bms_ble">
    Add the `esp32_ble_tracker`, `ble_client`, and `jk_bms_ble` blocks. The `throttle` parameter controls how often the component re-requests new data from the BMS.

    ```yaml
    esp32_ble_tracker:
      scan_parameters:
        active: false

    ble_client:
      - mac_address: ${mac_address}
        id: client0

    jk_bms_ble:
      - ble_client_id: client0
        protocol_version: ${protocol_version}
        throttle: 5s
        id: bms0
    ```
  </Step>

  <Step title="Add the BLE client switch for OTA compatibility">
    OTA updates require the BLE radio to be idle. Add the `ble_client` switch and configure the `ota` block to turn it off before flashing begins. Without this, an OTA update attempt will time out because the BLE client holds the radio busy.

    ```yaml
    ota:
      platform: esphome
      on_begin:
        then:
          - switch.turn_off: ble_client_switch0
          - logger.log: "BLE connection suspended for OTA update"

    switch:
      # … other switches …

      - platform: ble_client
        ble_client_id: client0
        name: "enable bluetooth connection"
        id: ble_client_switch0
    ```
  </Step>

  <Step title="Declare sensors and controls">

## Notes

<Info>
Reducing the `throttle` value below 5 s will increase data freshness but also increases BLE radio activity. Values below 2 s are not recommended and may cause connection instability on some BMS firmware versions.
</Info>

<Info>
The `number` platform entities (cell voltage limits, current limits, temperatures) write configuration registers back to the BMS over BLE. This bidirectional control is only available via the BLE connection; the UART-TTL connection is read/write at the modbus level but the `jk_bms` UART platform only exposes read-only sensors and on/off switches.
</Info>
